1. 按节点角色差异化裁剪（最直接的改进）

目前四个节点拿到的是同一份压缩结果。但它们的信息需求完全不同：

┌───────────────┬──────────────────────────────────────────────────────────┬────────────────────────────────┐
│ 节点          │ 真正需要什么                                             │ 现在多余的                     │
├───────────────┼──────────────────────────────────────────────────────────┼────────────────────────────────┤
│ planner       │ run.goal + 原始 data artifacts                           │ report/critique/plan artifacts │
├───────────────┼──────────────────────────────────────────────────────────┼────────────────────────────────┤
│ reporter      │ data artifacts + plan artifact + rewrite_reason + issues │ steps 细节、decisions          │
├───────────────┼──────────────────────────────────────────────────────────┼────────────────────────────────┤
│ critic        │ 最新 report artifact + decisions + issues                │ data artifacts 的 raw_response │
├───────────────┼──────────────────────────────────────────────────────────┼────────────────────────────────┤
│ evaluator     │ report + critique + metrics                              │ data artifacts                 │
└───────────────┴──────────────────────────────────────────────────────────┴────────────────────────────────┘

改进：在 compress() 增加 step_id 参数，按角色走不同的裁剪策略。

---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

2. 相关性感知的 artifact 保留（而非只保留最新）

目前只保留"最新一条"，但在 supplement 轮或 rewrite 轮中，历史 artifact 可能包含与当前 rewrite_reason 最相关的证据。

改进方向：

 - 对 artifact 做简单的关键词相关性打分（与 run.goal 和 rewrite_reason 做词汇重叠），保留相关性最高的而不是最新的
 - 对 report 类 artifact 保留最新2条（供 critic 比对前后轮的改进）

---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

3. 基于 token 估算而非字符数触发

当前用 len(json_str) 估算，对中文会低估（1个汉字 ≈ 1-2 tokens，但 JSON 序列化后字符数更大）。实际触发 LLM 摘要的时机可能偏晚。

改进：用 tiktoken 或模型 tokenizer 做精确 token 估算，阈值改为 token 数（如 6000 tokens）。

---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

4. 增量感知：只处理新增内容

每次调用 LLM 时，相比上一个节点，实际新增的 artifact/issue/decision 往往只有少量。

改进：

 - 记录上次压缩时的 artifact/issue ID 集合
 - 新的 artifacts 全量保留，已见过的 artifacts 只传摘要（或 ID 引用）
 - 类似"diff 模式"——让 LLM 看到"变化了什么"而不是"完整状态"

---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

5. LLM 摘要的质量验证

目前 LLM 摘要失败时直接 fallback 到截断，没有质量检验——如果摘要丢失了关键数字，后续节点得到的是劣质输入。

改进：

 - 对摘要做简单的数字/比率保留度检查（正则扫描原文中的数字，验证摘要中覆盖率）
 - 覆盖率不达标则保留原文截断版而非 LLM 摘要

---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

6. 结构化摘要替代自由文本摘要

目前 LLM 摘要输出是自由文本塞回 raw_response，会破坏下游节点对 parsed_response 字段的依赖。

改进：摘要的目标不是 raw_response（文本），而是重建一个精简的 parsed_response（结构化 JSON），让 reporter/critic 消费的仍是结构化字段，只是字段内容变短。

---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

7. 缓存摘要结果

同一个 artifact 在 planner → reporter → critic 三轮里会被摘要三次，是重复工作。

改进：以 artifact.id + 目标长度为 key 做内存缓存，同一 run 内复用。

---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

最优先落地的组合：方向1（节点差异化）+ 方向4（增量感知），能在不引入新 LLM 调用的前提下显著提升"传入信息的相关性"。方向6（结构化摘要）是中长期最有价值的，但需要对 artifact schema 做一定改造。
